### Integration Findings

**FINDING-01: Audit integrity service scope (AUDT-05, DFNS-04)**
- Integrity background service only registered when `result.active === true`
- Inactive-mode (no CANS.md) audit entries are written but not background-verified
- **Severity:** Design tradeoff — not a wiring break
- **Recommendation:** Confirm acceptable or move registration before the active-mode guard

**FINDING-02: Standalone entry point integrity (AUDT-05, DFNS-04)**
- `standalone.ts` does not create or register the integrity service
- `AuditPipeline.verifyChain()` is available for manual use by callers
- **Severity:** Acceptable for v1.0 (standalone is for CLI/library use)

**FINDING-03: Stale stub comments (code hygiene)**
- 11 `// Audit pipeline may be a stub` comments and try/catch blocks from Phase 1 persist
- Dead-defensive code — harmless but should be cleaned before Phase 4
- **Severity:** Informational
